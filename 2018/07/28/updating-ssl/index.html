Updating SSL

Jul 28, 2018 • Arnab Das • #100-words

After Chrome’s recent announcement, https for websites are becoming more mandatory. And it’s true that with the advent of Clouflare’s ambitious universal ssl and Let’s Encrypt using ssl has become affordable, after all it’s free. And there are lot of thirdparty tools to integrate ssl for popular web servers like Apache, Nginx.

I’m actually hosting my sites with ssl for quite a long time. Previously I used to buy cheapest Domain Verified certificates. After I moved to GitHub Pages + Cloudflare DNS combination, the need for buying SSL certificate also went away. It’s actually very easy and convenient. Though with this, the connection is not fully encrypted until the origin server as GitHub does not provide SSL for custom domains. So, from the client machine to Cloudflare’s load balancers it’s secure. But there to origin servers hosted by GitHub are not secure. As there is no transaction endpoint associated with my site, I’m not worried about this.

Though https has become the defacto of the web hosting industry, it’s fact that, secure connection has a overhead. And handshakes are slower compared to it’s plain counter part. And I’m also not understanding why this https is becoming this big agenda. It’s making the websites slowers, web servers more resource hungry. There is absolutely no need for https for simple blogs with some pages, like this one. And like my website, there are thousands of small blogs, using Cloudflare to serve “secure” content, while the connection to origin server not end-to-end encrypted. And browsers are not capable of detecting this flaw. Until I’m getting any solid reason to do this, I should say making every web site over ssl not only over cliched, but also just an “Eyewash”.
